# Codex 工作协议（给项目维护者与 Codex）

> 目的：让 Codex 既能理解 TraceDoc 的整体方向，又不会在一个小任务里把未来半年的功能一次性做掉。

## 一、这套协议解决什么问题？

把 Codex 当成一位执行速度很快、但需要明确施工图的工程师。

- 只告诉它“做一个文献系统”：它可能自行增加网页前端、向量数据库、Docker、LLM 等大量暂时不需要的部件。
- 只告诉它“写一个函数”：它可能不知道这个函数必须保留来源链，或者不知道原始 OCR 文本不能被覆盖。

因此每次工作都分成三层信息：

1. **README.md：项目总设计。**
   - 类似建筑总体规划。
   - 第一次接手项目时应完整阅读一次。
   - 后续任务只读任务书指定的章节即可。

2. **AGENTS.md：长期规则。**
   - 类似工地安全规范。
   - 每次工作都必须阅读。
   - 它规定哪些事绝不能做，例如：不要引入 Docker，不要覆盖原始 OCR 文本。

3. **任务书：本次施工许可证。**
   - 类似“今天只砌这一面墙”的图纸。
   - Codex 只能做任务书允许的内容。
   - 任务书必须写清目标、允许范围、禁止事项、验收条件和测试命令。

## 二、每次交给 Codex 的标准步骤

下面的流程可以直接复制给 Codex，也可以作为你自己检查任务的清单。

### Step 1：让 Codex 先理解任务，不要立刻写代码

建议提示：

```text
请先阅读：
1. AGENTS.md
2. 本次任务书：<任务书路径>
3. README.md 中任务书指定的章节

暂时不要修改文件、不要安装依赖。
先用自己的话列出：
- 本次任务目标；
- 不能违反的项目规则；
- 允许修改的文件；
- 明确不做的内容；
- 准备如何验证完成。
```

这样做的好处是：如果 Codex 理解错了，我们可以在写代码前发现，而不是等它提交一大堆不合适的文件。

### Step 2：确认理解后，才让它实施

建议提示：

```text
你的理解正确。现在只实施任务书中的内容。
请遵守 AGENTS.md，不要自行扩大范围。
完成后运行任务书指定的检查，并汇报：
1. 修改了哪些文件；
2. 新增了哪些依赖；
3. 运行了哪些命令及结果；
4. 哪些事情明确留到以后再做。
```

### Step 3：用独立分支和 Pull Request（PR）交付

每一个小任务使用一个分支。

示例：

```text
feat/p0-1-project-bootstrap
feat/p0-2-gold-set-format
feat/p0-3-core-protocols
```

一个分支只做一个任务。不要把“项目骨架 + OCR + 数据库 + UI”混在同一分支。

完成后由 Codex 创建 PR，目标分支为 `main`。PR 描述要按任务书的验收表逐项说明。

> 初学者注释：
> - **分支（branch）**：一条独立的修改路线；在上面试验不会直接影响主线。
> - **main**：通常代表当前被认可、可继续使用的主线版本。
> - **PR（Pull Request）**：请求把分支的改动合并回 main 的审查页面。
> - **commit**：一次保存到 Git 历史中的、带说明的变更记录。

## 三、何时让 Codex 阅读完整 README？

### 应完整阅读 README 的情况

- 第一次加入项目；
- 要设计跨模块接口或数据模型；
- 修改核心不变量，例如来源定位、原始输出保存、页面路由；
- README 本身发生大改后。

### 不必每次完整阅读的情况

- 只做一个已经有任务书的小模块；
- 修复一个局部 bug；
- 补一个确定的测试；
- 修改文档或 `.gitignore`。

任务书会写明本轮应该读 README 的哪些章节。这样 Codex 有全局地图，但不会把地图误当成待办清单。

## 四、对“重型运行环境”的默认态度

本项目当前刻意选择轻量路线：

- 使用 Windows 原生 Python；
- 使用现有 Conda 环境或普通虚拟环境；
- 使用 Python 自带 SQLite；
- 第一阶段不需要 Docker、Oracle、PostgreSQL、Qdrant、Redis、WSL 或本地大模型；
- OCR 和 PDF 解析库也应在对应阶段才加入，不在项目创建时一次装齐。

“以后可能会用”不是现在加入依赖的理由。

## 五、任务书模板

以后新增任务时，复制此模板到 `docs/00_project_control/tasks/`，再按实际内容填写。

```markdown
# 任务编号：<例如 P0-2>

## 1. 给初学者的一句话说明

<这项工作在整个项目里解决什么问题？>

## 2. 阅读范围

开始前必须阅读：
- `AGENTS.md`
- 本任务书
- `README.md` 的第 <章节号> 节

## 3. 目标

<完成后用户能做什么？>

## 4. 允许修改

- <文件或目录>

## 5. 明确禁止

- <不要做的内容>

## 6. 依赖限制

- <允许新增的依赖；若没有，写“不得新增运行时依赖”>

## 7. 验收标准

- [ ] <可检查的结果 1>
- [ ] <可检查的结果 2>

## 8. 测试与命令

```text
<必须运行的命令>
```

## 9. 交付要求

- 在分支 `<branch-name>` 中完成。
- 创建指向 `main` 的 PR。
- PR 描述中说明：改动、依赖、测试结果、未做事项。
```
